## 1. クライアントメニューの挙動調整

### 現状

- `/user` 配下のサブメニュー（クライアントメニュー）は、前回の対応で「ハンバーガーボタンで折りたたみ可能」になりました。
- ただし、**初期状態が閉じている** 挙動になっているため、ユーザー目線だと少し使いづらい状態です。

### 要望

1. **初期状態ではクライアントメニューを表示した状態にしてほしい。**
   - 特にデスクトップ・ラップトップなど横幅がある画面では、
     - ページを開いたときに左側にクライアントメニューが出ている状態をデフォルトにしてください。
   - モバイル幅（例: `< md`）に関しては、必要であれば初期クローズでも構いません。
     - ただし、その場合でも **ハンバーガーボタンで確実に開閉できること** を確認してください。

2. **クライアントメニューが出ているときは、メインの画面をそれに合わせて縮めるようにしてほしい。**
   - 具体的には：
     - デスクトップなどでは「左にメニュー / 右にメイン」というレイアウトにして、
       - メニューが出ているときはメインコンテンツの領域が適切に狭まる（オーバーレイしない）。
     - メニューの幅ぶんだけ `margin-left` や `grid` レイアウトでメインをずらすイメージ。
   - すでにサイドバー用のレイアウト（`app/user/(protected)/layout.tsx` など）があると思うので、
     - その中で `sidebarOpen` のような状態管理と、メインエリアのクラス（幅・余白）を連動させてください。

### 実装の方向性（例）

- 対象ファイル候補：
  - `app/user/(protected)/layout.tsx`
  - クライアントメニューやサイドバーを描画しているコンポーネント（components以下）
- やることの例：
  1. ビューポート幅に応じた初期状態を決める
     - 例：`useEffect` + `window.innerWidth` で md 以上なら `sidebarOpen = true`、それ未満なら false。
     - もしくは、サーバー側では常に「開いている」前提で描画し、クライアント側で上書きする。
  2. レイアウトを `flex` や `grid` で構成し、
     - `sidebarOpen === true` の場合はメニューぶんの幅を確保しつつメインを縮める。
     - `sidebarOpen === false` の場合はメインを全幅に広げる。
  3. 既に実装済みのハンバーガーボタンで `sidebarOpen` をトグルし、
     - 開閉時にメインエリアのクラスも同時に切り替える。

## 2. 管理者画面の実装状況とログインID仕様の確認

ユーザーからの質問：

> 管理者画面は作成している？ログインIDはどうなってる？

これに答えられるように、コードと DOCS を元に以下を整理してください。

### 2-1. 管理者画面の実装状況を確認

1. ルーティング構造の確認：
   - DOCS や `app` ディレクトリから、管理者用のページを探してください。
   - 例：
     - `app/manage/login/page.tsx`
     - `app/manage/(protected)/admin/page.tsx`
     - あるいは `app/admin/(protected)/...` など
2. 実際の挙動：
   - 管理者ログインページからログインして、
     - 正しく admin 用ダッシュボード / 管理画面に遷移できるか。
   - ロールチェック（`admin` グループ or customRole）がどうなっているか。
3. 現時点でのステータスを、レポートにまとめてください。
   - 例：
     - 「ログイン画面とダッシュボードは存在し、admin ロールのユーザーのみアクセス可能」
     - 「ルートはあるが、実際の画面が仮実装」
     - 「まだ管理画面は仮ルーティングのみで実装されていない」など。

### 2-2. 管理者ログインIDの仕様を整理

ここでいう「ログインID」は、ユーザー視点での疑問：

- 管理者アカウントはどのように作成される前提なのか？
  - Cognito コンソールから手動作成？
  - seed スクリプトで自動作成？
- ロールやグループはどうなっているか？
  - 例：Cognito User Pool の `admin` グループに所属しているユーザーを管理者とみなす
- 初期管理者ユーザーはどこかのドキュメントに書かれているのか？
  - メールアドレスやユーザー名の命名ルールなど。

やってほしいこと：

1. `DOCS/SEED_DATA.md` や `DOCS/IMPLEMENTATION.md`、`DOCS/AUTH.md` などに関連情報がないか確認。
2. `lib/auth/cognito.ts` や `middleware.ts`、管理者ルートの `layout.tsx` などから、
   - 「どの条件を満たすユーザーを admin として扱っているか」を読み取る。
3. 読み取った内容をもとに、
   - 「管理者ユーザーの前提仕様」を簡潔にまとめてください。
   - もし「初期管理者をこう作成してください」という手順が明確でなければ、
     - DOCS に「管理者ユーザー作成手順」のセクションを追加してください（例：`DOCS/AUTH.md` か `DOCS/SETUP.md` に数行程度）。
     - 具体的なメールアドレスは環境依存なので、`admin@example.com` のようなプレースホルダでOKです。

※ 実際の Cognito コンソールにはアクセスできないので、「このプロジェクトのコードから推測できる仕様」を整理する形で構いません。
